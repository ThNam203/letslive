1. PROJECT OVERVIEW
================================================================================

"Lets Live" is a full-featured live streaming and video-on-demand (VOD) platform
built with a microservices architecture. It supports:

  - Live streaming via RTMP (OBS-compatible) with real-time HLS transcoding
  - Video-on-demand uploads with background transcoding
  - Real-time live chat via WebSocket
  - Direct messaging (1-on-1 and group conversations)
  - User profiles with follow/unfollow system
  - Notifications system
  - Google OAuth and email/password authentication
  - Multi-language support (English, Vietnamese)
  - Cross-platform clients (Web + Mobile)

4. BACKEND MICROSERVICES ARCHITECTURE
================================================================================

All Go services follow a consistent structure:

  service/
  ├── cmd/main.go          # Entry point
  ├── api/http.go          # Route definitions
  ├── handlers/            # HTTP request handlers
  ├── services/            # Business logic layer
  ├── repositories/        # Database access layer
  ├── models/              # Data models
  ├── migrations/          # SQL migration files
  ├── middlewares/          # HTTP middleware
  ├── gateway/             # Inter-service HTTP clients
  ├── config/              # Configuration management
  └── pkg/                 # Shared packages (discovery, etc.)

Inter-service communication uses synchronous HTTP/JSON calls with Consul
service discovery. Request IDs propagate across services for tracing.

4.5 TRANSCODE SERVICE (ports: 1935 RTMP, 7779 health, 8889 webserver)
--------------------------------------------------------------------------------

Purpose: RTMP stream ingestion, real-time HLS transcoding, VOD transcoding

Features:
  - RTMP server for OBS/streaming software ingestion
  - Real-time FFmpeg transcoding to adaptive bitrate HLS
  - Quality variants: 360p, 480p, 720p (configurable)
  - HLS segment file watching and MinIO upload
  - Webserver for .m3u8 and .ts file serving
  - Background VOD transcoding worker (poll-based job queue)
  - Thumbnail extraction from video

RTMP Connection Flow:
  1. Streamer connects via rtmp://host:1935/live/{stream_key}
  2. Stream key verified via User Service
  3. Livestream record created via Livestream Service
  4. Raw RTMP piped to FFmpeg for HLS transcoding
  5. HLS segments uploaded to MinIO in real-time
  6. On disconnect: livestream ended, VOD created, files cleaned up

HLS Output Structure:
  /{streamId}/
  ├── 0/stream.m3u8      (720p variant)
  ├── 1/stream.m3u8      (480p variant)
  ├── 2/stream.m3u8      (360p variant)
  ├── segments/           (.ts files)
  ├── index.m3u8          (master playlist)
  └── thumbnail.jpg

VOD Transcoding Worker:
  - Polls transcode_jobs table every 5 seconds
  - Downloads raw file from MinIO
  - Transcodes to HLS with adaptive bitrate
  - Uploads segments & playlists to MinIO
  - Extracts thumbnail
  - Updates VOD status to "ready"
  - Deletes raw file
  - Retry: max 3 attempts with exponential backoff

Key Files:
  - backend/transcode/rtmp/rtmp.go            : RTMP server
  - backend/transcode/transcoder/ffmpeg.go    : FFmpeg transcoding
  - backend/transcode/worker/worker.go        : VOD transcode worker

12. KEY DESIGN DECISIONS
================================================================================

  1. MICROSERVICES ARCHITECTURE
     Each domain (Auth, User, Livestream, VOD, Chat) is an independent service
     for independent scaling and deployment.

  2. API GATEWAY PATTERN
     Kong centralizes routing, authentication (JWT), rate limiting, and CORS.
     Services don't handle auth directly - Kong injects user claims.

  3. SERVICE DISCOVERY
     Consul enables dynamic service registration. Services find each other
     via Consul DNS rather than hardcoded addresses.

  4. CENTRALIZED CONFIGURATION
     Spring Cloud Config Server provides Git-backed, versionable configuration
     with hot reload capability across all services.

  5. OBJECT STORAGE
     MinIO (S3-compatible) stores HLS segments, user uploads, thumbnails.
     Decouples file storage from application services.

  6. SIMPLE JOB QUEUE
     PostgreSQL-based transcode job queue (transcode_jobs table) instead of
     a dedicated message broker. Sufficient for current scale, upgradeable
     to Kafka/RabbitMQ later.

  7. DUAL DATABASE STRATEGY
     PostgreSQL for structured relational data (users, streams, VODs).
     MongoDB for schema-flexible chat messages.

  8. REAL-TIME WITH REDIS PUB/SUB
     Redis enables multi-instance chat message distribution without tight
     coupling between WebSocket server instances.

  9. OBSERVABILITY-FIRST
     Full OTEL integration with Grafana/Loki/Tempo from the start.
     Request IDs propagate across all service boundaries.

  10. CROSS-PLATFORM CLIENTS
      Web (Next.js) and Mobile (Flutter) share the same backend API.
      Both implement the same feature set with platform-appropriate UX.

  11. ADAPTIVE BITRATE STREAMING
      FFmpeg transcodes to multiple quality variants (360p, 480p, 720p).
      Master HLS playlist enables client-side quality adaptation.

  12. COOKIE-BASED AUTH
      JWT tokens stored in HTTP-only secure cookies (not localStorage).
      Kong extracts tokens from cookies for API authentication.
